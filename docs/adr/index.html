Architecture Decision Records (ADRs) document significant decisions made during the design of this project. Each ADR follows the Six Thinking Hats methodology: White (facts), Red (UX/emotion), Black (risk), Yellow (value), Green (innovation), and Blue (process/coordination). Click any record to read the full rationale on GitHub.

Decision Records

0001

Standardize API Response Format

Adopt a consistent JSON envelope (version, request_id, data, meta, error) for all API responses to simplify client integration and distributed tracing.

⬜ White Hat · Facts Accepted
0002

User Experience Error Messages

Provide actionable, human-readable error messages with typed codes, natural-language suggestions, and links to documentation — rather than bare HTTP status codes.

🔴 Red Hat · UX & Emotion Accepted
0003

Security Hardening for Public Release

Catalogue and address 16 attack surfaces: SQL injection, open CORS, missing security headers, unbounded body size, pprof exposure, plaintext DB passwords in logs, and more.

⚫ Black Hat · Risk & Caution Accepted
0004

Performance Optimization

Apply sync.Pool slice reuse, 4 KB buffered NDJSON streaming, and in-place next-permutation algorithm — projecting 30-45 % throughput improvement for large inputs.

🟡 Yellow Hat · Value & Benefit Accepted
0005

WebSocket Streaming Protocol

Add a bidirectional WebSocket endpoint with start/stop control, configurable chunk sizes, sequence numbers, and real-time cancellation — enabling streaming of millions of permutations without memory blow-up.

🟢 Green Hat · Innovation Accepted
0006

Structured Observability with OpenTelemetry

Adopt log/slog as the structured logging foundation, X-Request-ID middleware for distributed tracing, and a Prometheus-compatible /metrics endpoint — while deferring full OTel SDK integration.

🔵 Blue Hat · Process & Control Accepted
